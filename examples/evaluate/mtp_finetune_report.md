# MTP 微调结果汇报 — 原生 MTP vs 微调 MTP（ALLaVA 域内）

**日期**：2026-06-13　|　**模型**：Qwen3.5-9B（多模态 verifier）　|　**方法**：投机解码 MTP 头微调

## 结论（TL;DR）

在 100k Qwen 蒸馏 ALLaVA 数据上微调 Qwen3.5-9B **自带的原生 MTP 头**后，在域内 ALLaVA
验证集上**全面超过原生 MTP**：

> **平均接受长度 +7.7%（2.92 → 3.14），实测吞吐 +6.1%（66.4 → 70.4 tok/s）。**

接受率提升直接转化为吞吐提升，且无需改动推理框架——微调头缝合回 verifier 后即可正常 serve。

## 结果

| 指标 | 原生 MTP | 微调 MTP | 提升 | 比值 |
|---|---:|---:|---:|---:|
| **平均接受 token / draft** | 2.916 | **3.142** | +0.226 | **1.077** |
| token 接受率 | 0.417 | **0.449** | +0.032 | 1.077 |
| **输出吞吐 tok/s** | 66.38 | **70.39** | +4.01 | **1.061** |
| 首位接受率 (first-pos) | 0.838 | 0.844 | +0.006 | 1.007 |

- 两路均 **128/128** 请求成功完成。
- 提升主要来自**中后段位置**：首位接受率原生 MTP 已达 0.84（接近上限、提升空间小），
  增益体现在 draft 更深位置的 token 被接受得更多 → 平均接受长度 +7.7%。

## 实验设置（保证公平、无数据泄漏）

| 项 | 设置 |
|---|---|
| 训练数据 | 100k Qwen3.5-9B 自蒸馏 ALLaVA（域内）；bf16 + LR 3e-5；checkpoint 按验证 loss 选 |
| 验证集 | 该 100k 数据的**后 10%（held-out，9972 条）**，训练未见过 → 测的是泛化，非记忆 |
| 对比口径 | 原生 MTP = serve 原始 verifier（自带 MTP 头）；微调 MTP = serve 缝合后的检查点（微调头）。**两路同方法 `qwen3_5_mtp`、同投机步数 7、同验证集，唯一差别是 MTP 头权重** |
| 评测样本 | 128 条多模态（图+文）请求，温度 0；指标取自 vLLM `/metrics` 投机解码计数器 |

## 关键工程产出

- **打通了"微调 MTP 头如何上线评测"这一环**：vLLM 无法直接挂载裸 MTP 检查点（MTP 头嵌在
  verifier 权重内），需用 `scripts/stitch_mtp.py` 将微调后的 MTP 层**缝合**回一份完整 verifier
  检查点再 serve。
- 一键评测脚本：`examples/evaluate/test_mtp_allava_orig_vs_trained.sh`（自动缝合 + 双路对比 +
  一致性自检：若 trained≈original 会告警，提示微调头未加载）。本次自检通过，确认微调头已正确生效。

## 说明与下一步

- 本结果为**域内（ALLaVA）**评测，是训练目标场景；通用域的遗忘检查（如 MMStar）尚未做，建议补测。
- tok/s 存在 ±数% run-to-run 噪声，结论以接受率指标（first-pos / 平均接受长度）为准。
- 后续可继续：① 通用域遗忘检查；② 与微调 DFlash draft 同台对比；③ 对首位（pos-0）加权进一步抠
  first-pos，争取把平均接受长度再往上推。
