# Qwen3.5 多模态投机解码加速汇报 — MTP / DFlash 接受率提升

- **日期**:2026-06-22
- **范围**:Qwen3.5-9B(多模态)+ Qwen3.5-122B-A10B(MoE,~10B active,多模态)
- **目标**:在 vLLM 上用**投机解码**(speculative decoding)加速两款 Qwen3.5 多模态 verifier 的推理,在**不改变模型输出质量**的前提下提升吞吐
- **方法**:两条投机路线 —— **MTP**(微调 verifier 自带的多 token 预测头)与 **DFlash**(独立草稿模型);以 **ALLaVA**(域内)+ **MMStar**(域外)评测,均 128 prompts、temp=0、无数据泄漏

---

## TL;DR(核心成果)

1. **投机解码对 122B 最高带来 ~2.9× 吞吐**(MTP vs 无投机 baseline),且为**无损加速**(草稿被 verifier 逐 token 校验,输出分布与原模型一致)。
2. **MTP 是两款模型上最强的投机方法**(接受率 + 吞吐均第一);**训好的 DFlash 是稳健的第二**,在高并发/吞吐场景有独立价值。
3. **我们把 MTP 接受率系统性做了上去**:
   - **9B**:域内平均接受长度 **+7.9%**、域外 **+1.4%**(经"权重汤"后**两域同时为正**,零额外训练)。
   - **122B**:域内平均接受长度 **+11.5%**、吞吐同步 **+7~10%**;域外回退控制在 **−2.4%**(优于早期方案)。
4. **关键手段**:自蒸馏训练数据 + 微调原生 MTP 头 + 精度配置(MTP bf16 / DFlash fp32+CE)+ 缝合上线 + 权重汤(WiSE-FT)+ 步权重 A/B 调优。
5. **122B 步权重 A/B 结论(本周)**:`β=0.6`(衰减权重)**优于** `β=1.0`(等权),两域接受率均更高 —— 正式训练采用 `β=0.6`。

---

## 1. 背景与目标

多模态大模型推理中,**解码阶段(逐 token 自回归)是延迟主体**。投机解码用一个"草稿"机制一次性提出若干候选 token,再由主模型(verifier)一次前向**并行校验**,接受多少就等于"一次的代价吐多个 token"——因此:

> **吞吐 ∝ 平均接受长度 L**(每步净吐 token 数)。提升 L = 直接提升吞吐,且因校验机制保证**输出与原模型逐 token 一致(无损)**。

我们的工作就是在 Qwen3.5-9B 与 122B 两款多模态 verifier 上,把草稿质量(= 接受率 = L)做上去,并打通 vLLM 部署。

## 2. 两条投机路线:MTP vs DFlash

| | **MTP**(Multi-Token Prediction) | **DFlash** |
|---|---|---|
| 形态 | 微调 verifier **自带的原生 MTP 头**(嵌在模型权重内) | **独立草稿模型**(文本侧,多模态能力经 verifier 隐藏态注入) |
| 草稿方式 | 自回归(逐步) | **并行**(一次前向出整块) |
| 优势上限 | **接受率天花板**(最高 L) | **吞吐天花板**(每步草稿更便宜) |
| 实测定位 | 两款模型上**最强** | **第二**,高并发更耐用 |

两者互补:本期主力是 **MTP**(接受率最高);DFlash 作为高并发/吞吐场景的备选持续优化。

## 3. 我们如何提升 MTP 接受率(核心方法)

1. **训练数据 = 模型自蒸馏(train-on-distribution)**
   - 草稿要学的是 verifier **实际会输出什么**,而非数据集的人工答案。于是用模型**自己**在 ALLaVA 图文上生成续写(temp=0)作为训练目标:9B 用 Qwen 蒸馏,**122B 用其自身蒸馏**(因 MTP 必须学这台 122B 的分布)。
   - 规模:10k → 50k → 100k,跨两台机器分片(SKIP/MAX)并行生成。
2. **微调"原生 MTP 头",不是从零训**:Qwen3.5 自带 MTP 头(即 `qwen3_5_mtp` 服务的那个),我们在域内蒸馏数据上**继续微调它**,起点高、收敛快。
3. **精度配置(踩坑后定稿)**:**MTP 必须 bf16**(fp32 会破坏 MTP 的 lm_head);**DFlash 用 fp32 + 交叉熵(CE)损失** —— CE 是 DFlash 首次跑赢原始版的决定性修正(此前 kl_div 降了 loss 却不动 top-1,而 temp=0 接受率只看 top-1)。
4. **缝合上线(stitch)**:微调后的 MTP 头嵌在 verifier 权重内,vLLM 无法直接挂载 → 用 `stitch_mtp.py` 把微调层**缝合**回完整 verifier 再 serve,自检确认微调头已生效。
5. **权重汤 / WiSE-FT(α 混权)**:把微调头与原生头按 α 混合 → **在保住域内增益的同时修复域外回退**(9B:域外 −4.6% → **+1.4%**),零额外训练。
6. **步权重 A/B 调优(`step-weight-beta`)**:`β=0.6`(衰减,权重偏早位)vs `β=1.0`(等权,相对偏后位)。**结论 `β=0.6` 胜**——因接受是**链式**的(深位只有在浅位都被接受时才轮到),把训练权重集中在**高杠杆的早/中位**比摊到很少够到的深位更划算。
7. **训练深度(steps)**:草稿步数 3 → **5**(serve 用 spec=7;steps=7 在全词表下显存溢出,5 为折中且仍比原来深)。
8. **无泄漏评测口径**:训练用前 90%、评测用同集**后 10% 验证尾**;spec=7、temp=0、各 128 prompts、128/128 完成;域内(ALLaVA)+ 域外(MMStar)双轨。

## 4. 关键工程改动(我们改了什么)

| 改动 | 解决的问题 |
|---|---|
| **122B 张量并行(TP)训练通路** | 122B ≈ 244 GB(bf16)无法数据并行(会整模复制)→ 改为张量并行(verifier TP=4 占 GPU0-3 + trainer 占 GPU4-7);heads=32 → **TP 只能取 {4,8}** |
| **MoE `intermediate_size` 修复** | 层配置构建读的是稠密的 `intermediate_size`,而 122B MoE 只有 `moe_intermediate_size` → 加 fallback,否则 122B 训练直接崩(MTP/DFlash 通用) |
| **蒸馏并发 `--concurrency`** | 原蒸馏串行(并发=1)→ 加序保线程池 + 跨机分片,蒸馏大提速 |
| **缝合自包含 + hardlink 省盘** | 122B 不再整拷 244 GB;评测脚本自动缝合 |
| **评测自动化** | `mtp_accept` 自动切验证尾(后 10%);新增**逐机 A/B 评测脚本** `eval_mtp_ab.sh`(每台机评一臂);**跨机 native 基线一致性自检**(确认两臂可比) |
| **部署/服务踩坑固化** | 路径 `/home` vs `/data` 适配、`pkill vllm`、`/health` 探活 `no_proxy`、`--allowed-local-media-path` 图像路径对齐 |

| 平均接受 / draft(MMStar 域外) | 2.822 | — | 2.694(−4.6%) |
| 首位接受率(域外) | 0.828 | — | 0.824(−0.5%,持平) |

**MTP 权重汤(WiSE-FT, α=0.5)—— 两域同时为正,零额外训练**

| 指标 | 原生 | 微调(α=1) | **权重汤(α=0.5)** |
|---|---:|---:|---:|
| 域内接受 | 2.916 | 3.142 | **3.147(+7.9%)** |
| 域外接受 | 2.822 | 2.694(−4.6%) | **2.861(+1.4%)** |

**DFlash(暖启 + CE + fp32 + LR3e-5,100k 蒸馏)**

| 数据集 | 原始 DFlash | 训好的 DFlash |
|---|---:|---:|
| ALLaVA 域内 平均接受 | — | **+37%**(2.45× 无投机 baseline) |
| ALLaVA 域内 吞吐 | — | **+20%** |
| MMStar 域外 平均接受 | — | **+15%**(无灾难性遗忘) |

> 关键修正:`CE 损失 + fp32 隐藏态 + LR3e-5` 把 DFlash 首位接受率 0.727 → 0.775,使其首次在**两域**均超原始 DFlash 且吞吐最快。

### 5.2 Qwen3.5-122B-A10B(MoE)

**MTP 步权重 A/B(50k 蒸馏,steps=5,spec=7,TP=4)—— 本周结论**

| 指标 | native | **A:β=0.6** | B:β=1.0 |
|---|---:|---:|---:|
| ALLaVA 平均接受 / draft | 3.066 | **3.419(+11.5%)** | 3.376(+10.1%) |
| ALLaVA 吞吐 tok/s | 34.3 | **36.82(+7.3%)** | 36.47 |
| ALLaVA 首位接受率 | 0.848 | 0.866 | 0.873 |
| MMStar 平均接受 / draft | 2.930 | **2.862(−2.4%)** | 2.764(−5.7%) |
| MMStar 首位接受率 | 0.831 | 0.828 | 0.827 |

> **`β=0.6` 在两个数据集的接受率均高于 `β=1.0`**,域外回退也更小(−2.4% vs −5.7%)。"等权能抬后位"的假设被推翻 → **正式训练采用 `β=0.6`**。两臂跨机 native 基线一致(3.066 / 2.930),确认 A/B 可比。

**整体加速(122B,ALLaVA,对无投机 baseline)**

| 方法 | 对 baseline |
|---|---:|
| **MTP**(最强) | **~2.9×** |
| 训好的 DFlash | ~2.6× |
| 原始 DFlash | ~2.3× |
| 无投机 baseline | 1.0× |

- **训好的 DFlash vs 原始**:ALLaVA 平均接受 **+30.2%**、MMStar **+8.4%** → 122B DFlash 蒸馏训练有效,但仍落后 MTP。
- **吞吐 ∝ 接受长度得到印证**:MTP 的 +10.1% 接受对应 +9.9% tok/s(近 1:1)。

## 6. 结论与下一步

**结论**
- 投机解码在 Qwen3.5 多模态上是**无损、可观**的加速(122B ~2.9× 吞吐),**MTP 为首选方法**。
- 我们把 MTP 接受率在 9B / 122B 上系统性提升:**9B 经权重汤两域皆正**;**122B `β=0.6` 域内 +11.5%、域外仅 −2.4%**。

**下一步**
1. **部署 122B MTP best(`β=0.6`)**:域内 +11.5% 接受 / +7~10% 吞吐。
2. **通用场景上权重汤(WiSE-FT)**:把 9B 已验证的 α 混权用于 122B,修复域外回退、不掉域内。
3. **部署深度由 serve-time spec sweep {3,5,7} 定**:实测逐位**条件接受率深位仍稳在 ~0.74**(连超训练深度的位置),说明深位仍产出 token,不应过早砍到 3。
4. **DFlash 高并发优化(预研)**:`D-Cut`(校验深度动态剪枝,无需重训、不改主模型)用于 DFlash 部署路径在高并发/MoE 场景的进一步提速。

## 附:评测口径(保证可信、无泄漏)

- **数据**:训练用蒸馏集前 90%,评测用**同集后 10% 验证尾**(模型未见)→ 测泛化非记忆;域外用 MMStar(从未训练)。
- **对比口径**:native = serve 原始 verifier(自带头);trained = serve 缝合后的微调头。**同方法 / 同投机步数 / 同验证集,唯一差别是头权重。**
- **规模**:各 128 prompts,全部 128/128 完成;temp=0(greedy);tok/s 单次有 ±几% 噪声,以 acceptance 为准。
- **自检**:跨机 native 基线一致性 + trained≠native(确认缝合头已加载)。
